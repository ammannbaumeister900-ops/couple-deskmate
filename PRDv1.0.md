# Codex 开发提示词：情侣联动桌宠 Demo

## 一、项目目标

请开发一个 **情侣联动桌宠 Demo**。

这是一个用于产品验证的原型，不需要做完整商业化系统。核心目标是演示：

> 在同一台电脑上模拟两位情侣用户，各自拥有一个 Q 版二次元情侣小人。用户 A 点击“摸头 / 抱抱 / 想你 / 加油”等互动按钮后，用户 B 的桌宠会立刻或延迟产生表情、动画和状态反馈。

第一版不做真实联网，不做手机端，不做完整 IM，不做账号系统。
重点是做出 **“两个人的桌面小人可以互相感应”** 的演示效果。

---

## 二、Demo 形态

### 运行方式

开发一个 **同机双窗口模拟 Demo**：

* 一个窗口代表「我」；
* 一个窗口代表「对象」；
* 两个窗口并排展示；
* 点击一侧的互动按钮，另一侧角色立即响应；
* 也可以模拟“专注 / 离线”状态，触发延迟接收或离线胶囊效果。

### 推荐技术栈

优先使用：

* React
* TypeScript
* Vite
* Tailwind CSS

如需要模拟桌面软件感，可使用普通 Web 页面完成，不必真实开发 Electron。
但 UI 和交互要表现出“桌面宠物常驻层”的感觉。

---

## 三、产品定位

产品不是普通桌宠，也不是聊天软件，而是：

> 面向情侣的低打扰桌面陪伴层。

核心体验是：

1. 让对象的小分身住在电脑里；
2. 不打扰工作和学习；
3. 用轻互动表达情绪；
4. 让异地情侣获得持续陪伴感；
5. 用共享小家和情侣小人制造礼物感、专属感和传播感。

---

## 四、目标用户

主要面向三类人群：

### 1. 异地恋大学生情侣

* 年龄 18–24 岁；
* 常用 Windows 笔记本；
* 经常学习、自习、远程陪伴；
* 希望有低打扰的陪伴方式；
* 不想一直聊天，但希望知道对方在。

### 2. 游戏情侣

* 愿意为皮肤、挂件、头像、CP 定制付费；
* 喜欢桌面常驻的小角色；
* 关注是否影响游戏和电脑性能；
* 对“对象的小人陪我打游戏”有兴趣。

### 3. 热恋期年轻白领

* 年龄 22–30 岁；
* 常见场景是上班、加班、远程办公、出差；
* 想表达关心，但不希望频繁打扰对方；
* 关注隐私、安全、工作模式和静音模式。

---

## 五、核心痛点

产品需要解决以下问题：

1. 异地情侣不能实时陪伴；
2. 聊天太重，容易打扰；
3. 普通桌宠缺少双人联动；
4. 情侣礼物同质化，缺少专属感；
5. 一方想表达关心，但不想催促、查岗或打扰；
6. 桌面宠物如果遮挡操作，会很快被关闭；
7. 情侣数据敏感，需要让用户感到安全和可控。

由于暂时没有素材，请使用占位方式实现。

要求：

* 左右两个角色要有情侣感；
* 角色可以有不同颜色主题；

至少实现以下角色状态：

| 状态  | 表现               |
| --- | ---------------- |
| 默认  | 站立 / 坐着 / 轻微呼吸动画 |

## 3. 低打扰常驻层

要体现这个产品不是“悬浮玩具”，而是“低打扰桌面陪伴层”。

每个桌面区域里，角色默认停靠在：

* 桌面边缘；
* 或任务栏附近；
* 或右下角。

需要提供以下开关：

### 工作模式

开启后：

* 角色缩小；
* 停靠到边缘；
* 动画减少；
* 不主动弹大卡片；
* 互动只显示小表情。

### 静音模式

开启后：

* 关闭所有音效；
* UI 上显示静音图标。

### 点击穿透模拟

由于 Web Demo 无法真实点击穿透，可以做一个开关：

* 开启后按钮文案显示为「点击穿透：开」；
* 角色透明度降低；
* 角色旁提示：“当前模拟点击穿透，不阻挡桌面操作”。

### 全屏自动收边模拟

提供一个按钮：

* 点击「模拟全屏」；
* 两侧角色自动滑到屏幕边缘；
* 只露出一个小脑袋或小标签；
* 再点击退出全屏，角色恢复。

---

## 4. 三层联动机制

这是核心功能。

需要实现三种接收逻辑：

### 第一层：实时感应

当用户 A 点击互动按钮时，用户 B 的角色立即响应。

互动按钮包括：

* 摸头；
* 抱抱；
* 想你；
* 加油。

示例：

用户 A 点击「摸头」：

* 用户 B 桌宠播放“被摸头”动画；
* 出现气泡：“小鱼摸了摸你的头”；
* 用户 B 消息区域新增一条轻消息；
* 用户 A 侧显示发送成功反馈。

---

### 第二层：延迟同步

当用户 B 处于「专注」状态时，用户 A 的互动不立即大动画打扰用户 B。

逻辑：

* 用户 A 点击互动；
* 用户 B 不弹大动画；
* 用户 B 侧只出现小红点或小气泡；
* 消息进入「待接收动作」队列；
* 当用户 B 退出专注状态后，一次性播放这些动作。

需要在 UI 中展示：

* 待接收数量；
* “退出专注后接收”的提示。

---

### 第三层：离线消息胶囊

当用户 B 处于「离线」状态时，用户 A 的互动被存为情绪包。

逻辑：

* 用户 A 点击互动；
* 用户 B 不即时响应；
* 用户 B 侧显示“离线胶囊 +1”；
* 当用户 B 切换为在线时，系统依次播放离线期间收到的互动；
* 每个胶囊展示来源、互动类型和时间。

胶囊文案示例：

* “小鱼在你离线时抱了抱你。”
* “小鱼给你留了一个加油。”
* “小鱼刚刚想你了。”

## 7. 轻消息卡

不要做完整聊天系统，只做轻消息卡。

消息卡类型包括：

* 互动消息；
* 晚安卡；
* 想你卡；
* 加油卡；
* 离线胶囊；
* 系统状态卡。

每条消息包含：

* 发送人；
* 接收人；
* 类型；
* 简短文案；

* 状态：已发送 / 待接收 / 已播放 / 离线胶囊。

消息区域最多展示最近 5 条，超过后自动滚动或覆盖。

---

## 8. 状态系统

每一侧用户可以切换状态：

* 在线；
* 专注；
* 离线。

状态影响接收逻辑：

| 接收方状态 | 行为           |
| ----- | ------------ |
| 在线    | 立即播放互动       |
| 专注    | 存入待接收队列，不强打扰 |
| 离线    | 存入离线胶囊，上线后播放 |

UI 上需要清楚展示状态：

* 在线：绿色小点；
* 专注：蓝色 / 紫色小点；
* 离线：灰色小点。

## 十三、核心逻辑要求

请重点实现 `sendInteraction(fromUser, toUser, type)` 方法。

逻辑如下：

```ts
function sendInteraction(fromUserId, toUserId, type) {
  const receiver = getUser(toUserId);

  if (receiver.status === "online") {
    // 立即播放
    createMessage({
      fromUserId,
      toUserId,
      type,
      status: "played",
    });
    triggerReceiverAnimation(toUserId, type);
    increaseSharedHomeScore(type);
  }

  if (receiver.status === "focus") {
    // 延迟接收
    createMessage({
      fromUserId,
      toUserId,
      type,
      status: "pending",
    });
    showPendingBadge(toUserId);
  }

  if (receiver.status === "offline") {
    // 离线胶囊
    createMessage({
      fromUserId,
      toUserId,
      type,
      status: "capsule",
    });
    showCapsuleBadge(toUserId);
  }
}
```

当用户状态从 `focus` 切换到 `online`：

```ts
playPendingMessages(userId);
```

当用户状态从 `offline` 切换到 `online`：

```ts
playCapsuleMessages(userId);
```

## 十七、额外要求

请注意：

1. 不要做成聊天 App；
2. 不要做成普通宠物养成游戏；
3. 不要做公共社区；
4. 不要过度复杂；
5. 第一版必须围绕“双人联动感应”；
6. UI 要有 Demo 展示感；
7. 代码要方便后续接入 Electron 或真实 WebSocket；
8. 所有功能先本地模拟即可；
9. 没有素材时，请用 CSS / SVG / emoji 生成占位角色；
10. 最终效果要让用户一眼理解：这是情侣之间互相感应的桌面小人。
